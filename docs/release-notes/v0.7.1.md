# v0.7.1 — Adapter-trimming UX cleanup

Release date: 2026-05-03.

A focused user-experience release on top of the v0.7.0 publication
snapshot. The biological logic, periodicity QC, and downstream
deliverables are unchanged — only the adapter-trimming surface that
users interact with.

```bash
python -m pip install 'mitoribopy>=0.7.1'
mitoribopy --version          # MUST print 0.7.1 or later
```

---

## What changed

### Auto-detection becomes the only path; kit names are output-only

Earlier releases let users select a library kit by name (`--kit-preset
illumina_truseq_umi`, `kit_preset:` in YAML, `kit_preset` columns in
the sample sheet and sample-overrides TSV). v0.7.1 removes that input
vector. Two principles drive the change:

1. **Auto-detection is the default and the only path.** The pipeline
   scans the head of every input FASTQ, identifies the matching adapter
   family, and applies the right cutadapt parameters automatically.
   For mixed-kit batches each sample is resolved independently.
2. **Kit names are still reported.** When detection succeeds the
   matched family (`illumina_smallrna`, `illumina_truseq`,
   `illumina_truseq_umi`, `qiaseq_mirna`) is recorded in
   `kit_resolution.tsv` (`detected_kit` and `applied_kit` columns) so
   reviewers can see exactly what the detector picked.

The escape hatches are simpler now too:

* `--adapter <SEQ>` (CLI) / `adapter: <SEQ>` (YAML / sample sheet): pin
  the 3' adapter when detection cannot identify the library.
* `--pretrimmed` (CLI) / `pretrimmed: true` (YAML / sample sheet):
  declare already-trimmed FASTQs so cutadapt skips the `-a` flag.

The two are mutually exclusive: a sample is either getting an adapter
trimmed or already pre-trimmed.

### Migration

| Before (v0.7.0)                        | After (v0.7.1)                                                                  |
|----------------------------------------|---------------------------------------------------------------------------------|
| `--kit-preset auto`                    | omit (auto-detection is the default)                                            |
| `--kit-preset illumina_smallrna`       | `--adapter TGGAATTCTCGGGTGCCAAGG`                                               |
| `--kit-preset illumina_truseq`         | `--adapter AGATCGGAAGAGCACACGTCTGAACTCCAGTCA`                                   |
| `--kit-preset illumina_truseq_umi`     | `--adapter AGATCGGAAGAGCACACGTCTGAACTCCAGTCA --umi-length 8 --umi-position 5p`  |
| `--kit-preset qiaseq_mirna`            | `--adapter AACTGTAGGCACCATCAAT --umi-length 12 --umi-position 3p`               |
| `--kit-preset pretrimmed`              | `--pretrimmed`                                                                  |
| `--kit-preset custom --adapter <SEQ>`  | `--adapter <SEQ>` (no preset name needed)                                       |

YAML configs and sample sheets follow the same mapping. The
`kit_preset` column in the sample sheet, the sample-overrides TSV, and
the orchestrator's `align.samples:` YAML block was removed; use
`adapter:` / `pretrimmed:` instead.

`mitoribopy migrate-config` strips a `kit_preset:` value with a logged
warning so legacy configs can still parse — but the user is asked to
add an explicit `adapter:` (or `pretrimmed: true`) where the previous
named preset implied a sequence.

### `--adapter-detection off` semantics

The mode now requires `--adapter` (or `--pretrimmed`). Previously it
accepted `--kit-preset`; that path is gone.

### Removed internals

* `KIT_PRESET_ALIASES` and `mitoribopy.align._types.resolve_kit_alias`
  — they only existed to canonicalise legacy vendor names supplied by
  users. Internal kit names (`illumina_truseq`, `qiaseq_mirna`, …) are
  the only ones the detector emits.
* `SampleOverride.kit_preset` — replaced by `pretrimmed: bool | None`.
* `SampleRow.kit_preset` — replaced by `pretrimmed: bool | None`.
* `run_settings.json` field `kit_preset_default` — replaced by
  `pretrimmed_default: bool` alongside the existing `adapter_default`.

### Unchanged

* `KIT_PRESETS` registry (still drives detection and reporting).
* `kit_resolution.tsv` schema and column order.
* Periodicity QC, dedup behaviour, RPF / rnaseq stages.
* The TACO1-KO polyproline-stalling regression test still passes.

---

## Why

Two recurring papercuts on real datasets motivated this:

1. **Tutorial friction.** New users had to scan a kit-preset table to
   pick a name that matched their library. The detector already knew
   the right answer — adding the user-facing kit-name input vector was
   redundant.
2. **Inconsistent precedence.** When `--kit-preset` and `--adapter`
   disagreed the resolver had to pick a winner, and the rules were
   different across `auto` / `off` / `strict` modes. With `--adapter`
   as the only explicit input, the precedence rules collapse to "user
   wins; detector reports what it saw."

Because kit names still flow into `kit_resolution.tsv` for provenance,
publications and methods sections that cite the matched adapter family
keep doing so without changes.
