# v0.6.0 — Publication-readiness freeze (refactor-4)

This is the manuscript-target release. Per the 2026-05-01
publication-readiness audit, v0.6.0 closes the *interface coherence*
gaps that made earlier builds look less mature than the underlying
science. There are **no new biological features** and no changes to
numerical output — every difference is in how the package presents
itself to a reviewer or external programmer.

If you are pinning a version for a manuscript, pin to **v0.6.0**.
Older builds (≤ v0.5.1) shipped a different rpf flag style and a
now-removed no-subcommand fallback, and will not produce a config
that round-trips through `mitoribopy migrate-config` cleanly.

```bash
python -m pip install 'mitoribopy>=0.6.0'
mitoribopy --version          # MUST print 0.6.0 or later
```

## Removed: pre-subcommand fallback

Invoking `mitoribopy <flags>` without an explicit subcommand
previously routed to `mitoribopy rpf` with a one-line deprecation
warning. The warning incorrectly claimed the fallback would be
removed in v0.4.0 — it never was. v0.6.0 removes it for real.

The dispatcher now exits with code 2 and a clear message that points
at the right subcommand:

```text
mitoribopy: error: missing subcommand. Got '-s' as the first token,
but a subcommand is required as of v0.6.0. Did you mean
'mitoribopy rpf -s ...'? Run 'mitoribopy --help' for the full list.
```

Always use an explicit subcommand: `mitoribopy rpf …`,
`mitoribopy align …`, `mitoribopy all …`.

Underscore aliases continue to parse for one transition cycle and
will be removed in v0.7.0. They are no longer surfaced in `--help`,
in the README, in the generated CLI reference, or in
`mitoribopy all --dry-run` plans.

The rpf parser also reports as `mitoribopy rpf` (was bare
`mitoribopy`), `--config` accepts YAML / JSON / TOML (was JSON only
in `--help`), and the example block uses subcommand syntax.

YAML keys remain snake_case unchanged.

## Added: `mitoribopy all --strict`

A single switch that turns on every publication-readiness gate at
once. Rather than juggle three independent flags
(`validate-config --strict`, `--strict-publication-mode`,
`validate-figures --strict`), one invocation now does:

* runs `validate-config --strict` up-front and aborts before any
  stage if a deprecated key, unknown key, or missing input is found,
* forwards `--strict-publication-mode` into the align stage so
  ambiguous adapter detection, inferred-pretrimmed kits, legacy
  preset names, and the `--allow-count-invariant-warning` debug
  flag all fail-fast,
* runs `validate-figures --strict` after the pipeline finishes,
  promoting warn-only QC findings to fail in `figure_qc.tsv`.

`mitoribopy all --strict` is the recommended invocation for any run
that backs a manuscript figure.

## Added: `<output>/canonical_config.yaml` is now a first-class output

Every successful `mitoribopy all` run now writes
`<output>/canonical_config.yaml` (no flag required). It is the
fully-resolved config the run actually executed: legacy-key rewrites
applied, sample-sheet expansion materialised, `rnaseq_mode` resolved,
auto-wired `<output>/align/...` paths inlined. Reviewers can diff it
against the user-authored `pipeline_config.yaml` to see exactly how
the orchestrator interpreted the request.

The same blob is still embedded under `run_manifest.json`'s
`config_canonical` field. `mitoribopy all --print-canonical-config`
continues to write to stdout for use in pipelines that want to
inspect the canonical config without committing to an `--output`.

## Changed: orchestrator emits hyphenated argv for every stage

Pre-v0.6.0, `mitoribopy all` had to special-case the rpf section
with `flag_style="underscore"` because the rpf parser declared its
flags with underscores. With the rpf surface modernised, every stage
is now serialised through the same hyphen path. The
`flag_style="underscore"` mode is retained in `_dict_to_argv` for
back-compat tests but is no longer used by any first-party call site.

## Added: `docs/generate_cli_reference.py`

A small script that imports every subcommand parser and writes
`docs/reference/cli.md` from `parser.format_help()`. Run with
`--check` in CI to fail when the tracked file drifts from the live
parsers — `--help` text and the published CLI reference are now
guaranteed to agree.

## Docs

* README leads the quick start with the unified top-level `samples:`
  sheet (one TSV pairing Ribo-seq and RNA-seq by `sample_id`).
* The publication-safe recipe is rewritten around the new `--strict`
  switch.
* Tutorials and the `examples/templates/run_*.example.sh` scripts
  are rewritten to use hyphenated rpf flags.
* `docs/reference/cli.md` is now generated; `docs/reference/config_schema.md`
  spells out the YAML-snake-case ↔ CLI-hyphen-case rule explicitly.
* The `--allow-count-invariant-warning` flag is now labelled
  DEVELOPER / DEBUG ONLY in its `--help` line, and is rejected when
  combined with `--strict-publication-mode`.

## Compatibility

| What | v0.5.1 | v0.6.0 |
|---|---|---|
| `mitoribopy <flags>` (no subcommand) | warned, then routed to rpf | exit 2, "missing subcommand" |
| `mitoribopy rpf --offset_type 5` | accepted, shown in --help | accepted (legacy alias), not in --help |
| `mitoribopy rpf --offset-type 5` | unrecognized argument | canonical |
| `mitoribopy all --strict` | unknown flag | new top-level Strict Mode switch |
| `<output>/canonical_config.yaml` | only via `--print-canonical-config` | always written on real runs |
| Help line for `--allow-count-invariant-warning` | "Use only when debugging…" | "DEVELOPER / DEBUG ONLY" + rejected by `--strict-publication-mode` |

YAML configs from any v0.4.x or v0.5.x release still load — pass
them through `mitoribopy migrate-config` to canonicalise before
committing them to a manuscript repo.
