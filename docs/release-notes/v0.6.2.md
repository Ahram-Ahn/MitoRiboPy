# v0.6.2 — Fourth-edit cleanup

Release date: 2026-05-01.

This is the **fourth-edit cleanup release** on top of v0.6.0. It is the
manuscript-target version. Every change is software-engineering hygiene
(reproducibility, packaging, naming, docs, QC artefacts); no biological
logic was changed. Pin to **v0.6.2** for any run that backs a manuscript
or shared dataset.

```bash
python -m pip install 'mitoribopy>=0.6.2'
mitoribopy --version          # MUST print 0.6.2 or later
```

---

## Highlights

### Top-level `execution:` resource-plan block

Set the run-wide thread / parallelism / memory budget in one place at
the top of `pipeline_config.yaml`:

```yaml
execution:
  threads: auto                 # int | auto
  memory_gb: auto
  parallel_samples: auto
  single_sample_mode: false
  min_threads_per_sample: 2
  estimated_memory_per_sample_gb: 4.0
  scheduler: local
```

These cascade into stage-specific keys (`align.threads`,
`align.max_parallel_samples`, `align.single_sample_mode`,
`align.memory_gb`, `rnaseq.align_threads`) when those are unset; explicit
stage values still win. The orchestrator-level CLI flag
`mitoribopy all --threads N` now populates `execution.threads`, so
`mitoribopy all --threads 20 --config pipeline_config.yaml --output results/`
does what users always expected: 20 is the global budget.

### `resource_plan.json` at the run root

Every real `mitoribopy all` run now writes
`<output>/resource_plan.json` recording the resolved thread budget,
per-sample worker count, scheduler, single-sample mode, and the reason
the auto scheduler chose those values. The same blob is embedded under
`run_manifest.json -> resource_plan`. The manifest schema bumps to
`1.3.0` to advertise the new field.

### Periodicity QC bundle extension

`mitoribopy.analysis.periodicity_qc` adds a publication-grade bundle on
top of the existing per-length tables:

```
results/rpf/qc/
  qc_summary.tsv                       # one row per sample
  qc_summary.md                        # human-readable companion
  frame_counts_by_sample_length.tsv    # per-(sample, length) details
  gene_periodicity.tsv                 # per-(sample, gene) details
  by_length/
    frame_by_length.tsv                # inclusion-policy diagnostics
    length_inclusion_decisions.tsv     # distilled include/exclude verdict
    periodicity.metadata.json          # frame formula + thresholds
    frame_by_length_heatmap.png        # heatmap visualisation
```

Default thresholds (override in YAML):

| Knob                   | Default |
| ---------------------- | ------- |
| `good_frame_fraction`  | 0.60    |
| `warn_frame_fraction`  | 0.50    |
| `min_reads_per_length` | 1000    |
| `min_reads_per_gene`   | 50      |

QC labels: `good`, `warn`, `poor`, `low_depth`. The optional
`--phase-score` flag adds a ribotricer-style consistency column;
`calculate_fft_period3_ratio` is exposed for users who want the
period-3 power ratio as a supplementary metric.

### Canonical `dedup_strategy: umi_coordinate`

The publication-facing name is now the statistical operation, not the
implementation tool:

```yaml
align:
  dedup_strategy: umi_coordinate   # auto | umi_coordinate | skip
```

Legacy `umi-tools` / `umi_tools` keep parsing — `migrate-config` rewrites
them to `umi_coordinate`, and `canonical_config.yaml` always records the
canonical name. The implementation still calls into `umi_tools dedup`;
only the public name changed.

### `allow_pseudo_replicates` template key fix

The shorter key now serialises correctly to the real CLI flag
`--allow-pseudo-replicates-for-demo-not-publication` via
`migrate-config`. Templates lead with the long canonical key
`allow_pseudo_replicates_for_demo_not_publication:`; the short alias is
preserved for back-compat.

### Stable warning / error code registry

[`docs/reference/warning_codes.md`](../reference/warning_codes.md) is now
the user-facing index of every code emitted via
`mitoribopy.io.warnings_log`. The Python-side registry
(`src/mitoribopy/io/warning_codes.py`) is the single source of truth,
and the test suite refuses to record an unknown code.

### Release checklist

[`docs/developer/release_checklist.md`](../developer/release_checklist.md)
is now a hard gate for every PyPI cut. The README quick-install table
also pins to **v0.6.2**.

## Other changes

* Public module docstrings no longer carry "Phase 6 of v0.3.0 refactor"
  / "Phase 3.6 decision" annotations. The history is preserved in
  [`docs/developer/architecture_history.md`](../developer/architecture_history.md).
* `outputs_index.tsv` advertises the new outputs: `resource_plan`,
  `canonical_config`, `umi_qc`, `periodicity_qc_summary` (+ `_md`,
  `frame_counts_by_sample_length`, `gene_periodicity`,
  `frame_by_length`, `length_inclusion_decisions`,
  `periodicity_metadata`), and the per-site
  `codon_correlation_metrics` / `codon_correlation_metadata` pair.
* `mitoribopy periodicity` is a new standalone subcommand that re-runs
  the publication-grade periodicity QC bundle from a saved P-site /
  A-site coordinate table without re-invoking the rpf pipeline.
* `mitoribopy validate-config` recognises `execution:` as a known
  top-level section. Strict mode still rejects truly unknown keys.

## Migration from v0.6.0 / v0.6.1

There is no required migration. Existing configs continue to work
unchanged; `migrate-config` silently rewrites `dedup_strategy: umi-tools`
and `allow_pseudo_replicates:` into the canonical names.

The manifest schema bump (1.2.0 → 1.3.0) is additive. Downstream
scripts that gate on `schema_version` should accept anything in
`1.x.0`.

## Reverting

If a downstream environment cannot use the v0.6.2 features, pin to
`mitoribopy==0.6.0` — the v0.6.0 interface is still fully supported.
