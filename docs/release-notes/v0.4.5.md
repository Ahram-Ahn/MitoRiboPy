# v0.4.5 — picard dedup removed, RPF length auto-filter, compact COVERAGE log

Three independent improvements:

## Removed: `mark-duplicates` (picard) dedup option

Coordinate-only deduplication is biologically wrong for ribo-seq. Identical reads at the same transcript coordinate represent multiple ribosomes occupying the same codon — that is the **signal**. Without UMI information there is no way to distinguish those genuine biological duplicates from PCR artefacts, so collapsing them flattens the canonical polyproline stall and the TACO1-KO phenotype to baseline (see [docs/validation/taco1_ko_regression.md](../validation/taco1_ko_regression.md) for the empirical evidence that motivated the removal).

The v0.4.4 release gated `mark-duplicates` behind a 50-character confirmation flag (`--i-understand-mark-duplicates-destroys-mt-ribo-seq-signal`). v0.4.5 takes the next step and removes the option entirely:

- `--dedup-strategy` choices are now `{auto, umi-tools, skip}`.
- The `--i-understand-mark-duplicates-destroys-mt-ribo-seq-signal` confirmation flag is gone.
- The `picard` external-tool dependency is gone (no longer scanned at startup, no longer mentioned in the install instructions).
- Default behaviour is unchanged: `auto` resolves to `umi-tools` when UMIs are present and `skip` otherwise.

If you have an unusual non-ribo-seq library where coordinate-only dedup is appropriate, run picard `MarkDuplicates` manually on the `aligned/<sample>.mapq.bam` produced by the align stage, then feed the deduped BAM into `mitoribopy rpf` directly.

## Added: `--rpf_min_count_frac` auto-filter

Setting a fixed RPF window like `--rpf 28 34` keeps every read length in that window, including bins whose total count is a tiny fraction of the dominant length's count. Those low-count bins produce noisy or spurious offset picks because there are not enough reads at that length to resolve the canonical 12-15 nt 5'-end peak.

The new `--rpf_min_count_frac FRAC` flag (default `0.20`, on by default) automatically prunes read-length bins whose total count across all samples is below FRAC × the most-enriched length's count. The kept lengths are logged once with the threshold and the dropped lengths:

```
[BED] read-length auto-filter: kept [29, 30, 31, 32] (>=20% of max=1,234,567 reads); dropped [27, 28, 33, 34, 35, 36].
```

Set `--rpf_min_count_frac 0` to disable. The recommended pattern is to pair this with a wider `--rpf` range than the strict canonical window (e.g. `--rpf 27 36` for human) and let the data decide which lengths actually carry signal.

The filter operates on the combined-across-samples count so the surviving read-length set is consistent between samples; per-sample offset selection still varies within that shared set.

## Changed: compact `[COVERAGE]` log

The per-transcript log inside `run_coverage_profile_plots` previously emitted one long line per transcript per requested site (≈26 lines for human mt with `--analysis_sites both`). v0.4.5 replaces that with a single summary line per site after the loop:

```
[COVERAGE] P-site density: 13 transcript(s) plotted (rpm + raw + codon-binned + frame-coloured).
[COVERAGE] A-site density: 13 transcript(s) plotted (rpm + raw + codon-binned + frame-coloured).
[COVERAGE] read coverage: 13 transcript(s) plotted (rpm + codon-binned).
[COVERAGE] All coverage-profile plots generated.
```

Total: 4 `[COVERAGE]` lines per run instead of one per transcript per site.

## Migration

- Drop `--dedup-strategy mark-duplicates` and `--i-understand-mark-duplicates-destroys-mt-ribo-seq-signal` from any saved scripts; the default `auto` produces correct behaviour.
- The new `rpf_min_count_frac: 0.20` default is on for everyone; if you want the v0.4.4 behaviour add `rpf_min_count_frac: 0` to your YAML or `--rpf_min_count_frac 0` on the CLI.
